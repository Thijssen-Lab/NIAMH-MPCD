Outputs#

The MPCD code contains various input flags which can give different outputs. Default is that the output files are turned off (0), but they can be turned on by flagging them as a non-zero value. This non-zero value is the interval between outputs (in simulation timesteps). These different outputs can be classified as:

Synopsis#

The file synopsis.dat contains the entire synopsis output. This is the only output file that is writen as the default. It is also written at the start and finish of the simulation. Highly recommended to be on. 1 = on, 0 = off.

"synopsisOut":   1,

Fields#

Local course-grained data which varies spatially over MPCD boxes. This is outputted as time, then the cell position QX, QY, QZ (lattice integers) and then course-grained values.

The file coarsegrain.dat contains the field values for the coarse grained cell current population and current instantaneous velocity data.

"coarseOut":     0,

The file flowfield.dat contains the field values for the time-averaged velocity data (averaged over the timewindow between outputs).

"flowOut":       0,

The file velfield.dat contains the field values for the current instantaneous velocity data.

"velOut":        0,

The file directorfield.dat contains the field values for the local nematic director and scalar order parameter.

"dirSOut":       0,

The file ordertensor.dat contains the field values for the local Q tensor values.

"qTensOut":      0,

The file recipOrder.dat contains the field values for the local reciprocal space order parameter. In this output file, the QX, QY, QZ give the wave vector.

"qkTensOut":     0,

The file enfield.dat contains the field values for the local orientation interaction energy (if system is liquid crystal) and kinetic energy of the MPC fluid. Not MD or swimmers.

"oriEnOut":      0,

The file multiphase.dat contains the field values for the local species type of MPCD particle (which was previously referred to as colour in the literature), and multiphase phi parameter.

"colourOut":     0,

The file pressure.dat contains the field values for the local streaming pressure (pre-collision) and collision pressure (after collision) of the MPC fluid. It does not include MD monomers or swimmers.

"pressureOut":     0,

The file topochargefield.dat contains the local topological charge and orientation of the nematic fluid. Only works in 2D.

"topoFieldOut":  0,

The file disclinTensorfield.dat contains the three-dimensional disclination tensor data based on [Schimming2022].

"disclinOut":    0,

Trajectories#

Trajactories Gives the position QX, QY, QZ, the velocity VX, VY, VZ and the orientation OX, OY, OZ and rotational velocity LX, LY, LZ of various objects with the same idea as it evolves in time.

The file solidtraj.dat contains the trajectory of all moveable solid boundaries. Every obstacle gets their own file designated with a suffix.

"solidTrajOut":  0,

The file detailedSP.dat contains the trajectory of all the particles of species defined in trajSpecOut (the number indicates how many species one needs to write out). A user might only want to output the detailed trajectories of a small number of particles (perhaps as tracer particles) and not the majority of the MPCD fluid particles. The input trajSpecOut allows users to do this.

trajOut then specifies how often to output the detailed MPCD particle trajectories for every particle of species type given by trajSpecOut. Every species gets their own file designated with a suffix.

"trajOut":       0,
"trajSpecOut":   0,

The file defects.dat contains the position of topological defects found from grouping and averaging the non-zero topological field values and then tracking them. Function outputs coordinates, charge and angle of defect. Only works in 2D.

"defectsOut":    0,

The file swimmers.dat contains the position and velocity of all swimmers. It also contains if the particle is in the run, tumble, shring or extension phase.

"swimQOut":      0,

The file swimmersOri.dat contains the orientation of all swimmers. It also contains if the particle is in the run, tumble, shring or extension phase.

"swimOOut":      0,

Averages#

Averages. Simulation averaged date over the entire box. This is outputted as time and then averaged value.

The file avVel.dat contains the average current instantaneous ensemble-averaged velocity. It also outputs the system-averaged stress components.

"avVelOut":      0,

The file energy.dat contains the field values for the local nematic, kinetic and boundary energy contributions. The thermal energy and rotational energy at the boundary are also output.

"energyOut":     0,

The file avS.dat contains the system-averaged nematic scalar order parameter.

"avSOut":        0,

The file densSTD.dat contains the system-averaged local standard deviation of the density of all the cells.

"densSDOut":     0,

The file avEnstrophy.dat contains the system-averaged enstrophy (mean squared vorticity) of all the cells.

"enstrophyOut":  0,

The file enneighbours.dat contains the total energy based on the local director compared to its neighbours (i.e. the Frank deformation energy).

"neighbourEnOut":    0,

Histograms#

Outputs the probability distribution of a field at a certaint time. The number of bins is set by BINS (set to 101) in definition.h. Range is 0 (or minimum if more relavant for the exact field) and maximum field value.

The file distVel.dat contains the distributions of the velocity components.

"histVelOut":    0,

The file distSpeed.dat contains the distribution of speed (magnitude of velocity).

"histSpeedOut":  0,

The file distVort.dat contains the distributions of the vorticity components.

"histVortOut":   0,

The file distEnstrophy.dat contains the distribution of enstrophy magnitude.

"histEnsOut":    0,

The file distDir.dat contains the distribution of director components.

"histDirOut":    0,

The file distS.dat contains the distribution of nematic scalar order parameter.

"histSOut":      0,

The file distDens.dat contains the distribution of number of particles in each cell.

"histNOut":      0,

Autocorrelaion#

Output contains the spatial autocorrelation function of a current instantaneous field in radial coordinates with unit spacing in MPCD units.

The file corrVelVel.dat contains spatial autocorrelation for the field velocity.

"velCorrOut":    0,

The file corrDirDir.dat contains spatial autocorrelation for the director field.

"dirCorrOut":    0,

The file corrVortVort.dat contains spatial autocorrelation for the vorticity field.

"vortCorrOut":   0,

The file corrDensDens.dat contains spatial autocorrelation for the density (number of mpcd particles per cell)

"densCorrOut":   0,

The file corrOrderOrder.dat contains spatial autocorrelation for the nematic scalar order parameter.

"orderCorrOut":  0,

The file corrPhiPhi.dat contains spatial autocorrelation for the phase field.

"phaseCorrOut":  0,

The file energySpectrum.dat contains Fourier transformation of the velocity autocorrelation function (energy spectra). The steps are wavenumber.

"energySpecOut":  0,

The file enstrophySpectrum.dat contains Fourier transformation of the vorticity autocorrelation function (enstrophy spectra). The steps are wavenumber.

"enstrophySpecOut":  0,

Miscellaneous#

The file binderCumulant.dat contains the binder cumulant over the instantaneous orientation field (velocity if isotropic fluid; "lc": 0) for bin size defined in binderBin [Binder]. The output rate is set by binderOut and the bin size used is set by binderBin

"binderOut":     0,
"binderBin":     0,

The file runtumble.dat contains the signed angle which is the swimmers current orientations makes with their initial orientations, everytime it ends its tumbling/running phase. It also contains the time it is has spend tumbling. Set to 0 (off) or 1 (on)

"swimRTOut":    0,

Checkpointing#

The MPCD simulator has support for simulation checkpointing. This feature will store a complete dump of the internal simulation state to a file, which can then be loaded at a later time. Two particularly important use cases for this are:

  1. Chaining together simulations — If you need to run a system for a very long time, but cannot submit a single job to a cluster to run it, then you can run one system to a reasonable limit, checkpoint it, and then start a new simulation from the checkpoint after a break.

  2. Restarting simulations — If a simulation crashes, you can restart it from the last checkpoint, rather than starting over from the beginning.

Warning

A checkpoint file is not gauranteed to be compatible with future versions of the MPCD simulator. You should keep track of which version of the code you ran a simulation with and use the same version to restart a checkpointed run.

Launching a Simulation with Checkpointing#

There are two methods to launch a simulation with checkpointing enabled. These involve setting a specific input tag in the simulation input .json file:

  • "checkpointOut": <int> — After every <int> simulation steps have passed (across both warmup and normal runtime), the simulation will write a checkpoint file.

  • "checkpointTimerOut": <float> — After every <float> hours, the simulation will write a checkpoint file.

These options will override the other, so choose one or the other. Generally, checkpointOut is best for making a general backup to resume the simulation, whereas checkpointTimerOut is best for making a backup to chain together simulations.

Note

When later loading a checkpoint, it will remember the input you launched the simulation with during the original run. This means that if you want to chain together runs, set your simulation timestep count to be very large initially.

Warning

Checkpoint files store the entire simulation state, meaning that they can be very very large. MPCD itself is relatively stable, so more than one or two checkpoints per week of simulation runtime is probably overkill on a regular cluster.

Restarting a Checkpointed Simulation#

To restart a checkpointed simulation, move the checkpoint file checkpoint.dat to the same directory as your input .json file. In your input .json file, you are required to set the seed input tag to -1:

{
    "seed": -1
}

With this, launch MPCD normally and the simulation will resume from the checkpoint file.

Note

When loading a checkpointed simulation, MPCD will automatically load the input file you used to run the simulation originally.

Warning

The random number generator state vector is not saved during the checkpoint process, instead it is re-initialized using a pseudo-random seed. This means that if you restart two simulations from the same checkpoint, they can give different results from then on.
